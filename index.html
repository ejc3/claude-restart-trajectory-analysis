Detailed Analysis

01

Token Usage Variance

Despite originating from the same conversation session, there's a 36% difference in input tokens. This suggests Claude Code dynamically manages context by selectively including/excluding content based on relevance.

  • Trajectory A: 151,897 tokens with full file embeddings
  • Trajectory B: 96,859 tokens with summarized content
02

System Reminder Accumulation

Both trajectories contain <system-reminder> tags, but Trajectory A has significantly more. These are injected by Claude Code and accumulate over time.

  • Background Bash notifications (4 concurrent npm servers)
  • TodoWrite tool usage reminders
  • Malware analysis guidance reminders
03

Conversation History Depth

The formatted exports reveal an interesting pattern: Request 2 has more turns but fewer lines/tokens.

  • Request 1: 1,769 lines, up to turn [141]
  • Request 2: 1,540 lines, up to turn [148]
04

Embedded File Content

A major contributor to token variance is embedded file content from Read tool calls.

  • Trajectory A: Full _index.tsx (~131KB React component)
  • Trajectory B: Content summarized or omitted

Conclusions

  1. Dynamic Context Management: API payloads vary significantly within the same session based on content relevance determination.
  2. System Overhead: Background process notifications and reminders add substantial token overhead that compounds over time.
  3. File Embedding Impact: Reading large files into context can increase token usage by 50%+ per request.
  4. Context Pruning: Claude Code prunes older or less relevant content to manage context window limits.
Practical Implication: For cost optimization in long-running sessions, consider periodic restarts to clear accumulated context. Be mindful of Read tool usage on large files, and monitor background process overhead.
